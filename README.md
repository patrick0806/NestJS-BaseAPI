## Requirements

- **Node.js >= 24** (LTS). Enforced via the `engines` field in `package.json`.
- npm 10+ (ships with Node 24).
- **PostgreSQL** (local install, Docker, or cloud) reachable via `DATABASE_URL`.

## How to run

This project uses [NestJS](https://docs.nestjs.com/).

So it is recommended that you know a little about this framework

To run this project in dev mode you need to follow these steps

### Install dependencies

```bash
 npm install
```

### Configure the database

Set `DATABASE_URL` in your environment (or a `.env` file at the project root). The default used by `src/config/env/index.ts` is:

```
postgres://postgres:postgres@localhost:5432/referer
```

Create the database if it doesn't exist yet:

```bash
 createdb referer
```

(or `psql -U postgres -c "create database referer;"`)

Then apply migrations and seed the database:

```bash
 npm run db:migrate
 npm run db:seed
```

### Run project

```bash
 npm run start:dev
```

### API Reference (Scalar)

Once the app is running, the API reference UI is served by [Scalar](https://scalar.com) at:

```
 http://localhost:3000/api/v1/docs
```

The OpenAPI document is generated by `@nestjs/swagger` and rendered by `@scalar/nestjs-api-reference` (Fastify adapter). Request and response schemas are derived from [Zod](https://zod.dev/) via [`nestjs-zod`](https://github.com/BenLorantfy/nestjs-zod), so the documentation stays in sync with the runtime validation.

### Health check

The `/api/v1/health` endpoint pings the database using a custom Terminus indicator (`DrizzleHealthIndicator`). The response includes a `database` key reporting the DB status.

### Run tests

```bash
 npm run test
```

### Lint

```bash
 npm run lint
```

## Database scripts

| Script | Description |
|---|---|
| `npm run db:generate` | Diff the Drizzle schema against the migrations folder and emit a new SQL migration. Run after you edit anything under `src/config/database/schema/`. |
| `npm run db:migrate` | Apply all pending migrations using `drizzle-orm/postgres-js/migrator`. |
| `npm run db:push` | Push the schema directly to the database without generating migration files (useful for prototyping). |
| `npm run db:studio` | Open Drizzle Studio (web UI) to browse the data. |
| `npm run db:seed` | Run the seeders in `src/config/database/seeds/`. |

# Project Structure

## The project code is all inside src folder and we have three principal folders config, modules, shared

### Config

Here we have config files, like the API reference or the Drizzle database config. The base structure is like

```bash
    /config
        /env
            index.ts
        /apiReference
            apiReference.config.ts
            defaultResponses.ts
        /database
            database.config.ts
            database.module.ts
            drizzle.config.ts
            /schema
                index.ts
                users.ts
            /seeds
                seed.ts
                users.seed.ts
            /migrations
                0000_*.sql
            /health
                drizzle.health.ts
```

### Modules

Here we have our business logic and unit tests. Modules are contexts of a module and every module has the same structure like

```bash
  /modules
    /contexts
        /create
            /dtos
                request.dto.ts
                response.dto.ts
            /schemas
                request.schema.ts
                response.schema.ts
            /tests
                create.spec.ts
            create.controller.ts
            create.service.ts
    moduleName.module.ts
```

### Shared

We have the common logic of the application and shared things like entities and repositories that we can use in any module. The structure is like

```bash
    /shared
        /constants
            apiTags.ts
            index.ts
        /decorators
            public.decorator.ts
            roles.decorator.ts
            index.ts
        /dtos
            error.dto.ts
            index.ts
        /exceptions
            validation.exception.ts
            index.ts
```

## Adding a new DTO (Zod)

1. Create (or extend) a Zod schema under `src/modules/<module>/contexts/<context>/schemas/`.
2. Wrap it with `createZodDto(...)` to produce the DTO class used by `@Body()`, `@Query()`, etc.
3. Add `@ZodResponse({ status, description, type: ResponseDto })` to the controller method to emit OpenAPI metadata.
4. The global `ZodValidationPipe` (registered in `app.module.ts`) will validate the request automatically and throw `ValidationException` (a Zod-aware subclass) on failure, which the existing `ValidationExceptionFilter` turns into the project's standard `ExceptionDTO` error response.
